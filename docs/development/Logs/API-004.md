# API-004 - Teste de autorização para criação de pedido

Data: 2026-03-28

## Objetivo
Concluir a task `API-004` da Fase 003, garantindo autorização server-side para criação de pedidos, bloqueio de tentativas em nome de terceiro e disponibilidade de logs de auditoria para tentativas indevidas.

## Escopo executado
- Nova policy de autorização para criação de pedido:
  - `src/server/application/security/create-order.policy.ts`
  - regras aplicadas:
    - `customer` só cria pedido para o próprio `userId`
    - `admin` pode criar para qualquer `customerId`
    - `organizer` e `checker` são bloqueados
- Handler com verificação de autorização no boundary autenticado:
  - `src/server/api/create-order.handler.ts`
  - request agora recebe `actor` autenticado
  - bloqueio antes de executar use-case
  - auditoria best-effort de tentativa indevida com payload seguro
- Export de security atualizado:
  - `src/server/application/security/index.ts`
- Cobertura de testes adicionada/atualizada:
  - `tests/unit/server/application/security/create-order.policy.test.ts`
  - `tests/unit/server/api/create-order.handler.test.ts`
  - `tests/integration/api/orders/auth.test.ts`
  - `tests/integration/api/orders/create-order.endpoint.integration.test.ts` (ajuste para `actor` autenticado)

## Evidências de aceite
- Critério: pedido em nome de terceiro é bloqueado
  - Evidência: `tests/integration/api/orders/auth.test.ts` valida `403` para `customer` com `customerId` divergente
- Critério: logs de tentativa indevida disponíveis para auditoria

## Observações
- A autorização foi mantida no boundary da API com policy em `application/security`, preservando handler fino e separação de responsabilidades.
- O log de auditoria é best-effort e não altera a resposta de autorização em caso de falha do logger.
