# DBR-003 — Repositórios Drizzle iniciais: decisões técnicas

**Data:** 2026-03-27
**Status:** ✅ Concluída

---

## Decisão central

O driver `drizzle-orm/neon-http` (via `@neondatabase/serverless` HTTP) **não suporta `db.transaction()`**. Chamadas ao método lançam `Error: No transactions support in neon-http driver` em runtime.

Por isso, `DrizzleOrderRepository.create` persiste pedido e itens com **inserts sequenciais sem atomicidade DB-level**.

---

## Contexto

A arquitetura atual usa:

```
@neondatabase/serverless → neon() → drizzle(sql, { schema }) via drizzle-orm/neon-http
```

Essa combinação é a escolha correta para **Cloudflare Workers** (runtime serverless sem TCP persistente), mas o endpoint HTTP da Neon não executa múltiplas queries dentro de um `BEGIN/COMMIT` explícito via Drizzle.

---

## Consequência imediata

`OrderRepository.create` insere `orders` primeiro, depois `order_items`. Se o segundo insert falhar:

- A linha `orders` persiste sem itens (órfã).
- O use-case lançará exceção, mas o pedido incompleto fica no banco.

Para a fase atual (demo, sem tráfego concorrente), o risco é baixo e aceitável.

---

## Caminhos para produção

### Opção A — Driver WebSocket (`neon-serverless`)

Trocar o client para:

```ts
import { Pool } from "@neondatabase/serverless";
import { drizzle } from "drizzle-orm/neon-serverless";

export function createDb(databaseUrl: string) {
  const pool = new Pool({ connectionString: databaseUrl });
  return drizzle(pool, { schema });
}
```

O driver WebSocket suporta `db.transaction()` completo. Cloudflare Workers suporta WebSockets nativamente. Trade-off: latência de primeiro request ligeiramente maior (handshake WS).

### Opção B — Compensação na camada de aplicação

Manter o HTTP driver e, no use-case `createOrder`, incluir cleanup explícito em caso de falha parcial (ex: deletar o pedido órfão se o insert de itens/tickets falhar). Mais verboso, mas sem dependência de upgrade de driver.

### Opção C — Neon Batch API (raw SQL)

O objeto `sql` do `@neondatabase/serverless` expõe `sql.transaction([...queries])` que envia múltiplas queries como batch HTTP transacional. Usável contornando Drizzle para a operação crítica, mas perde type-safety do ORM.

---

## Decisão para fase 003

Antes de implementar o handler de criação de pedido (API-002), avaliar Opção A. A migração de driver é cirúrgica (apenas `client.ts` + tipo `Db`) e desbloqueia transações sem rewrite de repositórios. Detalhes em `PHASE-003 API-002`.

## Notas adicionais

- `fileParallelism: false` adicionado ao config de integração porque múltiplos arquivos de teste compartilham o mesmo banco Neon e causavam `TRUNCATE` concorrentes com FK violations.
- `testTimeout: 15000` configurado para absorver cold-starts do Neon serverless (conexões HTTP frias podem levar 5–10 s).
